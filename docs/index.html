Current state

What you can run today.

C is the test-backed execution path

Source lowers through lexer, parser, type checker, MIR, and C99 output before a native C compiler produces the executable. C is the only backend covered by the 868-test native execution baseline.

HLSL and GLSL shader output are working

Shader examples emit readable HLSL/GLSL from the same QuantaLang source.

Other backends are research surfaces

Rust, LLVM IR, WebAssembly, SPIR-V, x86-64, and ARM64 are wired into quantac build --target, but they are not presented as equal in maturity to the C path.

Evidence

How to judge it.

Run a program

Build quantac, run hello.quanta, ledger.quanta, and effects_greeting.quanta, then verify the 8-program semantic corpus. The point is not a perfect language ecosystem; it is a real compiler path with receipts nearby.

Use the shader path

Compile vignette_shader.quanta to HLSL or GLSL when you want readable shader output from Quanta source.

Read the limits

STATUS.md says the self-hosted compiler and standard library exist as .quanta source, but cannot be compiled or executed today.

Progress and aspiration

Make room for both.

Where it is now

Today, QuantaLang is a Rust-built effects compiler with a verified C execution path, working HLSL/GLSL shader-source output, typed effect receipts, policy gates, SourceId provenance, and semantic-corpus checks.

Where it is trying to go

Long term, QuantaLang is meant to become a live-state-aware language substrate: one source shape that can coordinate CPU and GPU outputs, declare machine and model boundaries, and emit receipts that WARDEN-style membrane tooling can inspect.

Compiler workflow

How the compiler works.

A .quanta source moves through lexer, parser, type checker, MIR, and backends. C is the test-backed execution path; shader output writes source; the remaining backends are labeled by maturity. That boundary is part of the project: ambition, but with the current state kept visible.

01

Algebraic effects

Effect declarations, perform, handle, and handler dispatch are in the quickstart path.

02

Native execution through C99

quantac run compiles through generated C, with fixtures for functions, loops, traits, dispatch, patterns, effects, and color/math examples.

03

vignette_shader.quanta and graphics fixtures emit reusable HLSL/GLSL math.

04

Status checks and diagnostics

quantac corpus verify checks the 8-program semantic corpus and C baseline. Experimental Rust smoke slices stay separate. quantac doctor reports backend readiness.

One source shape

One language family for CPU and shader code.

CPU examples run through C. Shader examples emit HLSL/GLSL. Speculative targets stay labeled research-grade. This is the same instinct as the rest of my work: build the artifact, then name exactly what it can and cannot do.

effects_greeting.quanta
effect Greeter {
  greet(name: str) -> str;
}

fn say_hello(name: str) -> str {
  perform Greeter.greet(name)
}

fn main() {
  handle say_hello("Alice") with Greeter {
    greet(name) => "Hello, " + name + "!"
  }
}
vignette_shader.quanta
#[fragment]
fn PS_Vignette(uv: vec2) -> vec4 {
  let color = tex2d(uv);
  let vig = vignette(color.x, color.y, 0.5, 0.6);
  vec4(color.x * vig, color.y * vig, color.z * vig, 1.0)
}

Live-state provenance substrate

Effects are a boundary language.

Why this matters

LLMs can hallucinate the state they describe; compilers and machines can report the state they actually touched. QuantaLang points toward code that declares effects, records ambient capability use, emits CPU and shader artifacts with maturity labels, and hands those receipts to WARDEN-style live-state tooling.

Where it could apply

The same pattern belongs in creative tools, research pipelines, and security workflows: code should say whether it touched files, network, GPU, process, environment, foreign functions, or model-facing boundaries before a human or model trusts the output.

Current boundary

This is an aspiration, not a finished platform claim. C remains the verified execution path today; HLSL/GLSL are working shader-source outputs; simultaneous CPU/GPU orchestration with WARDEN is the direction, not the current release promise.
